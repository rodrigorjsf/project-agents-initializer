Evidence-based instructions for structuring `.cursor/rules/*.mdc` hierarchies in Cursor.

## Rule Placement Decision Table

When deciding where content belongs, use this table:

| Activation mode | Use when content is... | Load timing |
|-----------------|------------------------|-------------|
| `alwaysApply: true` rule | Critical tooling or constraints needed on **every single task** | Always (every conversation) |
| `globs:`-mode rule | Specific to certain file patterns | On-demand when files match |
| `description:`-mode rule | A cross-cutting / domain topic the agent should pull in by name | On-demand when the agent decides |
| Skill | A workflow the agent should invoke explicitly | On-demand when invoked |
| Domain doc referenced from a rule | Reference content a rule cites instead of inlining | On-demand via the rule |

**Borderline tiebreaker (10–15 lines):** When content falls between 10–15 lines, prefer extracting it to a separate `description:`-mode rule if it applies to only one topic. Always-loaded rule brevity is more valuable than the small cost of an additional rule.

*Source: Industry Research — a-guide-to-agents.md lines 228-233; research-context-engineering-comprehensive.md lines 257-305*

## Always-Loaded Rule Content

Generate `alwaysApply: true` rules with **only** these elements:

1. **Non-default tooling commands** — package manager, build/test/lint scripts that override conventional names
2. **Hard project-wide constraints** — invariants the agent must preserve on every task
3. **Non-default config overrides** — strict type-checking, line-length overrides, coverage addopts

> "Consider this the absolute minimum... That's honestly it. Everything else should go elsewhere."
> — Industry Research, a-guide-to-agents.md

Anything that does not pass the always-loaded bar belongs in a `globs:`-mode or `description:`-mode rule.

## Monorepo: Package-Scoped Rules

In monorepos, prefer narrow `globs:`-mode rules scoped to package paths over broad `alwaysApply: true` rules. Always-loaded rules add tokens to **every** conversation, regardless of which package the agent is working on.

| Rule scope | Generate here | Avoid here |
|------------|---------------|------------|
| Project-wide always-apply | Cross-package tooling commands the orchestrator runs (workspace install, root build) | Per-package conventions |
| `globs:`-mode rule per package | Package-specific tech stack quirks, dependency rules, exports patterns | Cross-repo decisions |

> "Don't overload any rule scope. The agent sees all merged rules in its context."
> — Industry Research, a-guide-to-agents.md lines 164-193

**Monorepo context scoping in Cursor**: In large monorepos, use `.cursor/rules/*.mdc` with narrow `globs:` patterns to limit rules to relevant packages. Rules with `alwaysApply: true` load for every request — keep them minimal.

*Source: Industry Research — memory/how-claude-remembers-a-project.md lines 243-260*

## Progressive Disclosure Patterns

Apply these patterns when content exceeds always-loaded scope:

**Extraction trigger**: Move a section into its own `description:`-mode rule when it has 3+ distinct rules AND spans 10+ lines, OR when the topic is irrelevant to most work sessions (e.g., database migration conventions in a project where most work is UI changes).

**Move domain content out of always-apply rules:**

```markdown
# In an alwaysApply rule — do this:
For TypeScript conventions, see the typescript rule.

# NOT this (inline domain content):
Always use const instead of let. Never use var...
```

**Use skills for workflows** — agents invoke skills only when needed, keeping base context minimal.

*Source: Industry Research — a-guide-to-agents.md lines 110-163*

## Cursor Rule Scoping Notes

- Use `.cursor/rules/*.mdc` for every project convention; legacy monolithic context files are not generated by this distribution
- Keep rule frontmatter minimal: only `description`, `alwaysApply`, and `globs`
- Prefer narrow `globs:` over `alwaysApply: true` when guidance applies to specific files only
- Prefer `description:` over `alwaysApply: true` when the content is topic-attractor reference material the agent can pull in by name

## Anti-Patterns to Detect and Remove

| Anti-pattern | Why it fails | Fix |
|--------------|--------------|-----|
| Ball-of-mud growth | Each fix adds an `alwaysApply: true` line; after hundreds of additions the always-loaded surface confuses the agent | Refactor into `globs:`-mode and `description:`-mode rules |
| Auto-generated init files | "Prioritize comprehensiveness over restraint" — flood with irrelevant content | Author manually, restraint first |
| Everything in one rule | Exceeds ~150-200 instruction attention budget | Split by concern |

> "There's a natural feedback loop that causes context files to grow dangerously large."
> — Industry Research, a-guide-to-agents.md lines 34-43

---

## Validation

Before finalizing any generated hierarchy, check:

- [ ] `alwaysApply: true` rules contain ONLY: critical tooling and hard project-wide constraints
- [ ] Domain content lives in `description:`- or `globs:`-mode rules, not inlined into always-apply rules
- [ ] No file exceeds 200 lines
- [ ] Each rule covers exactly one concern
- [ ] Monorepo conventions live in `globs:`-mode rules scoped to package paths, not in `alwaysApply: true` rules
